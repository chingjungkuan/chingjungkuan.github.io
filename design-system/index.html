#B2B #Design System #Figma

Team | TinTint 點點印

Role | Owner & Contributor

Tools | Figma

Deliverables | DS + Token + Guideline

  • 背景：點點印服務橫跨官網與 Editor 2 兩大設計場景，過去缺乏統一的元件規範，導致跨頁面的視覺不一致與開發落差。

設計系統全貌 System Overview

Foundation 品牌語言決策

色彩系統 Color

主色 / 輔助色 / 中性色 / 語意色

整套色彩系統以品牌主色 #7B6453 為錨點，搭配 CTA 橘 #EB9164 作為行動呼籲色，並以 Basic 灰階（0–100）作為文字、背景、分隔線的基礎。Success / Info / Warning / Danger 語意色則確保狀態反饋語義一致。

為什麼這樣選？
  • 主品牌色選用溫暖大地棕，呼應點點印印刷質感與親和力定位。
  • CTA 橘與主色同色調但彩度更高，製造視覺對比又維持品牌一致性。
  • Danger 文字色獨立設定為深紅 #90341A，確保在白底上的對比度符合無障礙標準。

字型排版 Typography

字重階層（H1–H6）

字級從 H1（36px）收斂到 Badge（10px），統一落在 8px–36px 範圍。任何非標準尺寸（如 17px）視為 Design Smell，需重新審視設計需求。

行高 × 視覺節奏

按鈕文字分為 Large（18px）、Medium（16px）、Small（14px），每個尺寸同時提供 Bold 與 Regular，確保不同場景的資訊層次清晰可辨。

Spacing・圓角・陰影

4px / 8px Base Grid

所有 margin 與 padding 必須落在 4、8、12、16、20、24、32、40、48、64px 數列內，對應 Token 命名如 $spacing-1 = 4px。非此數列的間距視為規範外數值，需提出說明。

Corner Radius 獨立 Collection

圓角規格以獨立的 Collection 統一管理，確保全站元件的形狀語言一致，修改時只需調整一處即可全站同步。

陰影層次

Shadow-0 到 Shadow-2 對應不同元件層次（卡片、選單、浮層），Active Shadow 則用於 focus / active 狀態的外發光，讓互動反饋更直覺。

Design Token 四層架構

解耦（Decoupling）是核心

這套 Token 架構共分四個 Collection，每一層職責明確、單向引用。主題切換、深淺模式切換，皆透過變數層級自動驅動，不需手動改元件。

  • Primitive Color：儲存品牌色的完整色階（10–90），只定義「色彩是什麼」，禁止直接套用於 UI 元件或程式碼。
  • Theme / Brand：整套架構最核心的設計決策。將 Primitive Color 對應到 Primary 色階，並以 mode 支援多主題切換——換主題只改這層。
  • Semantic：賦予顏色使用情境的語意，分為 background、text、divider、solid btn、outline btn 等群組。同時定義 Light / Dark 兩個 mode，是開發端唯一需接觸的層級。
  • 元件：直接套用 Semantic Token，不寫死任何色碼。

進階思維：骨架固定，內容彈性

解決「一致性 vs 彈性」的拉鋸

發現問題，也找到解法

Figma 工具的架構限制，導致新增元件時往往需要堆疊大量 Variants，讓 DS 元件結構越來越肥大。Figma 推出 Slots 新功能後，剛好解決了這個卡點。因此從最常被 detach 的元件下手——Dialog、Modal、Card、Panel，讓這些元件大幅減少 Variant 需求，Library 也得以保持乾淨可維護。

Slots：骨架固定，內容可換

透過 Figma Slots 將元件拆為「不變的結構骨架」與「可替換的內容區域」，全程不需 detach。概念等同前端的 Vue <slot> 或 React children，讓設計端的結構真正對齊開發端。

最優先套用的元件

從最常被 detach 的元件下手——Dialog、Modal、Card、Panel，這些地方 Slots 能立刻大幅減少 Variant 需求，讓 Library 保持乾淨可維護。

DS 實戰延伸

頁面前情提要簡介

主題募集 2.0

品牌依不同檔期推出活動頁面，透過募集主題照片並搭配產品模型，打造互動式體驗。頁面需同時支援多套主題色與 Light / Dark Mode 切換，是驗證 DS 四層架構擴充能力的核心專案。

Occasion 2.0

整合原 Idea 頁與 Occasion 頁面，解決定位不明問題，重新定調為「情境式引導主題專區」。以對象、情境、預算為軸引導導購決策，涵蓋母親節、婚禮、情人節等多類情境，並預留領券功能版位。

官網頁面

配合品牌視覺升級，官網重新定調為現代清新文青風格，品牌色系隨之重塑。元件設計以色彩與規格的彈性延伸為前提，確保跨頁面的視覺一致性與長期可維護性。

設計決策

為什麼需要多主題 + 深淺色？

主題募集 2.0 需要同時支援「預設黃色系」與「客製色」兩套主題，以及 Light / Dark Mode 切換——這正是四層架構的最大壓力測試。

Theme / Brand 層的設計決策

新增 mode 即完成主題替換，完全不動 Semantic 層或元件。未來新增節慶主題，只需在 Theme 層加一組 Mapping，整套系統自動跟進。

Semantic 層的 Light / Dark

每個語意 Token 同時定義兩個 mode 的對應值。設計師寫 solidBtnBgPrimaryNormal，不需為深色另寫一組——切換工作交由底層 root 覆蓋。

色彩角色比例 60 / 20 / 10 / 10

多主題系統除了 Token 架構，還需要一套「頁面配色比例規則」來確保換了主題之後視覺不失控。OCC 2.0 實戰確立了四個色彩角色的職責與用量。

Neutral 中性色

頁面基底色，確保閱讀舒適度。Body Background、Card Background、內文。

60%
Primary 主色

活動主題色，代表當前活動氛圍。Header/Nav、主要按鈕、Outline 框線。

20%
Secondary 輔助色

主色同色系的協調陪襯色，豐富畫面層次但不搶戲。區塊背景、Coupon 小圖示。

10%
Highlight 點綴色

高飽和、高對比，用於打破畫面平衡抓取注意力。CTA、促銷標籤、Heading 強調字。

10%

關鍵守則：Highlight 須與 Primary 形成強烈對比；三個有彩色（Primary / Secondary / Highlight）不可同時高飽和，否則頁面失去視覺重心。

截圖建議：三欄並排驗證

最能說服面試官的呈現方式：

  • Primitive 面板 → Theme 面板（含 mode 欄） → Semantic 面板，三欄並排，直接展示單向引用關係。
  • 同畫面並排：同一個 UI 在 Light / Dark 兩個 mode 下的視覺對比，讓 Semantic 層的自動適配效果一目了然。

這套規範已整併回 DS 主庫，成為後續專案的正式規格基礎。

套用成果

一致性的可維護設計決策

Outcome

系統讓跨頁面、跨主題的品牌一致性成為可維護的設計決策——設計師思考的是語意，而不是色碼。

Before：假設無 DS
  • 各頁面元件各自為政，同一個按鈕出現三種不同色值。
  • Dark Mode 需手動為每個元件另寫一組樣式。
  • 換主題等於重畫整個設計稿。
After：DS 套用後
  • 主題切換只改 Theme 層一個 mode，全站自動更新。
  • Light / Dark 由 Semantic 層自動處理，無需額外工作。
  • 這套規範已整併回 DS 主庫，作為下一版本的基礎。

DS 讓設計師專注更上層的問題

當 DS 把「元件應該長什麼樣」變成既定規則，設計師的注意力才能放在「這個元件應該在哪裡出現、為什麼出現」。以 OCC 2.0 為例，DS 建立後設計端可以直接聚焦在導購邏輯與轉化率設計：

▍主題募集 2.0 實際產品畫面

設計師的職責不只是畫稿

這次建立 DS 讓我深刻體會：設計師在交付畫面之前，更重要的是建立「讓畫面可以被系統性維護」的底層結構。Token 架構設計得好，未來每一次迭代都能省下大量的溝通成本。

解耦思維：改一處，動全局

四層架構最核心的價值是解耦。主題切換、深淺色切換都不需要手動改元件，這種「改一處、動全局」的設計思維，讓系統在面對未來需求時具備真正的可擴張性。

命名即文件

語意化的 Token 命名（如 solidBtnBgPrimaryHovered）本身就是文件。工程師不需要額外查表，看見變數名就知道對應的元件、屬性與狀態，大幅縮短設計與前端的溝通落差。

真實專案才是最好的壓力測試

主題募集 2.0 讓我在真實的專案中驗證了 DS——當需求提出「要支援紫色主題」時，只新增一個 Theme mode 就搞定了。這證明架構設計的假設是正確的，而不只是理論上可行。

系統穩了，才能思考轉化

DS 建立前，每次改版都在修「按鈕顏色又跑掉了」這類問題。DS 建立後，設計端才有空間去思考「這個按鈕放在這裡，能不能提升轉化率」——這才是設計真正應該聚焦的事。
