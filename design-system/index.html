  • 背景：橫跨官網頁面與編輯器兩大設計場景，過去缺乏統一的元件規範，導致跨頁面的視覺不一致與開發落差。
  • 目標：從零建立一套可長期維護的品牌設計系統，涵蓋 Design Token 四層架構、Component Library，以及支援多主題與深淺色模式的擴充機制。
  • 成果：此作品集分為兩階段——DS 基礎建立，以及透過全官網頁面統一和主題募集 2.0 專案等進行的實戰延伸驗證。

Foundation 品牌語言決策

色彩系統定義 Color

以品牌主色 #7B6453 為視覺錨點，構建完整的色彩層級體系。中性色階（Basic 0–100）負責支撐全站文本、容器與佈局節奏；語意色則嚴格遵循行業標準，確保系統狀態反饋的視覺語義（Semantic Clarity）高度一致。

  • 品牌親和力：選用溫暖大地棕，呼應點點印具溫度的印刷質感與親和力定位。
  • 視覺權重優化：CTA 橘在維持品牌色調一致的前提下，透過高彩度對比強化行動呼籲的導購權重。
  • 系統性擴展：不同狀態色基於品牌色系進行階層化處理，在維持整體視覺和諧的同時，兼顧優異的辨識度與 Accessibility 考量。

字型排版 Typography

字級與資訊階層

字級範圍嚴格控制在 8px - 36px 之間。排除決策干擾，任何非標準尺寸（如 17px）均視為 Design Smell，需回歸規範審視其必要性，以維持介面高度的一致性。

行高規範

依據資訊類型定義行高規則：內文 (1.5) 以優化閱讀易讀性為前提；標題與標籤 (1.2) 則側重視覺凝聚力，確保跨組件的排版節奏穩定。

元件文本適配

針對按鈕與控制項，定義 Large / Medium / Small 三階字級並提供雙字重選擇，確保功能性文字在不同資訊密度下的資訊階層清晰可辨。

Spacing・圓角・陰影

4px / 8px Base Grid

所有 margin 與 padding 必須落在 4、8、12、16、20、24、32、40、48、64px 數列內，對應 Token 命名如 $spacing-1 = 4px。非此數列的間距視為規範外數值，需提出說明。

Corner Radius 獨立 Collection

圓角規格以獨立的 Collection 統一管理，確保全站元件的形狀語言一致，修改時只需調整一處即可全站同步。

陰影層次

Shadow-0 到 Shadow-2 對應不同元件層次（卡片、選單、浮層），Active Shadow 則用於 focus / active 狀態的外發光，讓互動反饋更直覺。

Design Token 四層架構

解耦（Decoupling）是核心

這套 Token 架構共分四層，每層職責明確、單向引用。主題切換、深淺模式切換，皆透過變數層級自動驅動。

  • Primitive Color：儲存品牌色的完整色階（10–90），只定義「色彩是什麼」，禁止直接套用於 UI 元件或程式碼。
  • Theme / Brand：整套架構最核心的設計決策。將 Primitive Color 對應到 Primary 色階，並以 Mode 支援多主題切換。
  • Semantic：賦予顏色使用情境的語意，分為 background、text、divider、solid btn、outline btn 等群組。同時定義 Light / Dark 兩個 Mode，是開發端唯一需接觸的層級。
  • 元件：直接套用 Semantic Token，不寫死任何色碼。

進階思維：骨架固定，內容彈性

解決「一致性 vs 彈性」的拉鋸

導入 Slots 機制，有效解決了以往元件庫因堆疊過量 Variants 而導致的維護負擔。優先針對 Dialog、Modal 與 Card 等高頻被取消關聯 (Detach) 的組件進行架構重整，在賦予設計極大自定義彈性的同時，也確保 Library 保持極簡且高效的品質。

DS 實戰延伸

官網頁面簡介

主題募集 2.0

品牌依不同檔期推出活動頁面，透過募集主題照片並搭配產品模型，打造互動式體驗。頁面需同時支援多套主題色與 Light / Dark Mode 切換，是驗證 DS 四層架構擴充能力的核心專案。

Occasion 2.0

整合原 Idea 頁與 Occasion 頁面，解決定位不明問題，重新定調為「情境式引導主題專區」。以對象、情境、預算為軸引導導購決策，涵蓋母親節、婚禮、情人節等多類情境，並預留領券功能版位。

全站頁面

配合品牌視覺升級，官網重新定調為現代清新文青風格，品牌色系隨之重塑。元件設計以色彩與規格的彈性延伸為前提，確保跨頁面的視覺一致性與長期可維護性。

設計決策

為什麼需要多主題 + 深淺色？

主題募集 2.0 需要同時支援「預設黃色系」與「客製色」兩套主題，以及 Light / Dark Mode 切換——這正是四層架構的最大壓力測試。

Theme / Brand 層的設計決策

新增 Mode 即完成主題替換，完全不動 Semantic 層或元件。未來新增節慶主題，只需在 Theme 層加一組 Mapping，整套系統自動跟進。

Semantic 層的 Light / Dark

每個語意 Token 同時定義兩個 Mode 的對應值。設計師寫 solidBtnBgPrimaryNormal，不需為深色另寫一組——切換工作交由底層 Root 覆蓋。

切換 Mode，全頁自動對應

左圖 Light、右圖 Dark——兩個畫面共用完全相同的元件實例，背景色、文字色、按鈕色全部由同一組 Semantic Token 驅動，只需在 Root 切換 Mode，無需複製或手動改色。

▍Token 實際運作

以頁面底色為例：bgPageNormal 這個 Semantic Token 在 Light Mode 對應 #FFFFFF、在 Dark Mode 對應 #121212，切換的工作完全由變數層處理，元件本身沒有任何修改。

▍Header・Navbar・Card・Footer 全部同步

頁面上的所有元件（導覽列、標籤列、產品卡、頁尾...）都套用 Semantic Token，不寫死任何色碼。「新增一個深色主題」的成本，從重複整份設計稿，降低到只需確認一次 Token Mapping 是否正確。

色彩角色比例 60 / 20 / 10 / 10

多主題系統除了 Token 架構，還需要一套「頁面配色比例規則」來確保換了主題之後視覺不失控。Occasion 2.0 實戰確立了四個色彩角色的職責與用量。

Neutral 中性色

頁面基底色，確保閱讀舒適度。Body Background、Card Background、內文。

60%
Primary 主色

活動主題色，代表當前活動氛圍。Header/Nav、主要按鈕、Outline 框線。

20%
Secondary 輔助色

主色同色系的協調陪襯色，豐富畫面層次但不搶戲。區塊背景、Coupon 小圖示。

10%
Highlight 點綴色

高飽和、高對比，用於打破畫面平衡抓取注意力。CTA、促銷標籤、Heading 強調字。

10%

關鍵守則：Highlight 須與 Primary 形成強烈對比；三個有彩色（Primary / Secondary / Highlight）不可同時高飽和，否則頁面失去視覺重心。

Occasion 2.0

Atomic 第 5 層的實戰落地

Occasion 2.0 主題情境頁是第五層最直接的體現，Navbar、Hero、Card、Coupon Ball、Bookshelf 等下層元件依導購策略組合，構成目的明確的解決方案頁面。

▍9 個區塊，每一層都有導購邏輯

Hero（分流入口）→ 情境引導（Scenario）→ 主題應用（Application）→ 產品比較 → 作品書架（靈感）→ Coupon → 製作流程 → 品牌特色 → FAQ，從吸引注意到協助決策，層層遞進。

▍元件跨裝置一致性驗證

各裝置區間皆明確定義元件的呈現方式，Announcement Bar、Navbar、Sticky Coupon Ball 等核心元件在四個斷點全部對應，無需為不同裝置重新設計元件。

Variables 面板：三層單向引用

▍Primitive → Theme → Semantic 不會反向污染

三個 Collection 單向引用，Semantic 層只取 Theme 層的值，Theme 層只取 Primitive 層的值——任何一層的修改，都不會影響上層結構。

套用成果

一致性的可維護設計

這套規範經過官網相關頁面、主題募集 2.0 與 Occasion 2.0 後，整併回 DS 主庫，成為後續專案的正式基礎。

▍DS 持續調整優化
  • 主題切換只改 Theme 層一個 Mode，全站自動更新。
  • Light / Dark 由 Semantic 層自動處理，無需額外工作。
  • DS 內容持續調整優化，以適應不斷變化的設計需求。
▍Before：假設無 DS
  • 各頁面元件各自為政，同一個按鈕出現三種不同色值。
  • Dark Mode 需手動為每個元件另寫一組樣式。
  • 換主題等於重畫整個設計稿。

DS 讓設計師專注更上層的問題

把「元件應該長什麼樣」變成既定規則，作為設計師的注意力即可放在「這個元件應該在哪裡出現、為什麼出現」。
以 Occasion 2.0 專案為例，DS 建立後設計端可以直接聚焦在導購邏輯與轉化率設計。

Takeaway

設計師的職責：從維護畫面到創造價值

建立 DS 之前，每次改版都在修「按鈕顏色又跑掉了」這類問題；建立之後，設計端才有空間思考「這個按鈕放在這裡，能不能提升轉化率」。Token 架構設計得好，不只省下溝通成本，更讓設計師的注意力回到真正應該聚焦的事——讓畫面可以被系統性維護的底層結構，才是交付畫面之前更重要的工作。

解耦思維：改一處，動全局

四層架構最核心的價值是解耦。主題切換、深淺色切換都不需要手動改元件，這種「改一處、動全局」的設計思維，讓系統在面對未來需求時具備真正的可擴張性。

命名即文件

語意化的 Token 命名（如 solidBtnBgPrimaryHovered）本身就是文件。工程師不需要額外查表，看見變數名就知道對應的元件、屬性與狀態，大幅縮短設計與前端的溝通落差。

真實專案才是最好的壓力測試

主題募集 2.0 讓我在真實的專案中驗證了 DS——當需求提出「要支援紫色主題」時，只新增一個 Theme Mode 就搞定了。這證明架構設計的假設是正確的，而不只是理論上可行。
